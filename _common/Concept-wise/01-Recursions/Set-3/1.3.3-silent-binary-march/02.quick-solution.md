# Silent Binary March — Quick Solution

## Idea (Simple)

We build the binary code one position at a time.  
At each position, we can always try placing `0`.  
We can place `1` only if the previous bit was not `1`, the position is not restricted, and the position is not the final position.  
So recursion must remember the current position and the previous bit.

## Pattern Recognition Clues

- The problem asks for the number of valid sequences, so this is counting recursion.
- Each position has repeated choices: place `0` or place `1`.
- A future choice depends on the previous bit because consecutive `1`s are forbidden.
- Restricted positions and final-position rule are constraints that must be checked before choosing `1`.

## State Discovery

The first simple state is:

```text
solve(pos)
```

This means the number of valid ways to fill positions from `pos` to `n`.

But `solve(pos)` is not enough.  
At the same position, the choice `1` may or may not be allowed depending on the previous bit.

The missing information is:

```text
prevBit
```

So the final state is:

```text
solve(pos, prevBit)
```

## Pattern

Counting Recursion with Previous Choice State

## Core Insight

- Placing `0` resets the consecutive-`1` risk.
- Placing `1` is allowed only after `0` and only on unrestricted non-final positions.
- The total answer is the sum of all valid recursive branches.

## Invariant

For every call `solve(pos, prevBit)`, all positions before `pos` have already been built validly, and `prevBit` correctly stores the bit placed at `pos - 1`.

## Algorithm (with one-line why at important steps)

1. Start at position `1` with previous bit `0`.  
   - Why: no real previous bit exists, so `0` safely allows the first `1` if valid.
2. If `pos > n`, return `1`.  
   - Why: all positions have been filled with valid choices.
3. Always try placing `0`.  
   - Why: `0` is allowed even on restricted positions.
4. Try placing `1` only if `prevBit == 0`.  
   - Why: this prevents consecutive `1`s.
5. Also check that the current position is not restricted.  
   - Why: restricted positions cannot contain `1`.
6. Also check that `pos != n`.  
   - Why: the final position must be `0`.
7. Add the valid branch counts.  
   - Why: counting recursion uses addition.

## Mental Cheat Sheet

| Item | Meaning |
|---|---|
| Pattern | Counting recursion with previous choice state |
| State | `(pos, prevBit)` |
| Choice | place `0` or valid `1` |
| Constraint | no consecutive `1`s, restricted positions, final `0` |
| Answer | `solve(1, 0)` |

## Complexity

- Time Complexity: `O(2^n)` in pure recursion.
- Space Complexity: `O(n)` due to recursion stack.
- Auxiliary Space: `O(n)` due to recursion stack.

## Tiny Dry Run

For:

```text
n = 3, restricted = []
```

The final bit must be `0`.

Valid codes are:

```text
000
010
100
```

The code `110` is invalid because it has consecutive `1`s.  
So the answer is `3`.

## Notes / Pitfalls

- Do not allow `1` at the final position.
- Do not block `0` at restricted positions.
- Do not use only `solve(pos)`. The previous bit changes the next valid choices.
- Counting recursion adds branch answers. It does not take `min` or `max`.
