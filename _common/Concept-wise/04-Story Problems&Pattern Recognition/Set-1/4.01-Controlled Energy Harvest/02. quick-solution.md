## Idea (Simple)
We have energy nodes in a line.
Normally, we cannot collect from two adjacent nodes.
But we are allowed to break this rule at most once.

So each node gives two choices:
- skip it
- take it

To decide whether taking is legal, we must remember two things:
- whether the previous node was taken
- whether the one adjacency override was already used

## Pattern Recognition Clues
- The problem asks for the maximum total energy, so this is optimization DP.
- Each node creates a take-or-skip choice.
- Taking the current node depends on whether the previous node was taken.
- The one-time override adds a small extra state: already used or not used.

## State Discovery
The first idea is `dp[i]`: best energy up to index `i`.
But `dp[i]` is not enough.
It forgets whether index `i` was taken.
That matters because index `i + 1` is adjacent to index `i`.

A second idea is `dp[i][usedOverride]`.
That is still not enough.
It tells whether the override was used, but it still forgets whether the previous node was taken.

So we use this state:
- `used`: whether the override has already been used
- `prevTaken`: whether the previous node was taken

This keeps exactly the missing information needed for the next decision.

## Pattern
Take / Skip Dynamic Programming with State Augmentation

## Core Insight
- If the previous node was not taken, the current node can be taken normally.
- If the previous node was taken, the current node can be taken only if the override is still unused.
- Skipping the current node is always legal.

## Invariant
After processing some prefix of nodes, `dp[used][prevTaken]` stores the maximum valid energy for that prefix, with:
- `used` showing whether the adjacency override has been used
- `prevTaken` showing whether the last processed node was taken

No state contains more than one adjacent selected pair.

## Algorithm (with one-line why at important steps)
1. Start with `dp[0][0] = 0`.
   - Why: before processing any node, no override is used and no previous node is taken.
2. Mark all other states as impossible.
   - Why: they do not represent any real selection yet.
3. For each node value, create a fresh `next` table.
   - Why: each node updates the state for the next prefix.
4. Try skipping the current node.
   - Why: skipping is always valid and makes `prevTaken = 0`.
5. Try taking the current node when `prevTaken = 0`.
   - Why: there is no adjacency conflict.
6. Try taking the current node when `prevTaken = 1` only if `used = 0`.
   - Why: this spends the one allowed adjacency override.
7. After all nodes, return the maximum over all four states.
   - Why: the best answer may or may not use the override, and the last node may or may not be taken.

## Mental Cheat Sheet
| Item | Meaning |
|---|---|
| Pattern | Take / Skip DP |
| Missing information | previous taken status + override status |
| `used = 0` | override still available |
| `used = 1` | override already spent |
| `prevTaken = 1` | last processed node was selected |
| Answer | maximum over all final states |

## Complexity
- Time Complexity: `O(n)`
- Space Complexity: `O(1)`
- Auxiliary Space: `O(1)`

## Tiny Dry Run
For `energy = [2, 7, 9, 3, 1]`:

One strong valid choice is:
- take node `7`
- take adjacent node `9` using the override
- take node `1`

Total = `7 + 9 + 1 = 17`.

The DP accepts this because only one adjacent pair is created: `7` and `9`.
It rejects choices that would create two adjacent selected pairs.

## Notes / Pitfalls
- Do not use only classic House Robber recurrence.
- Do not allow a chain like three consecutive taken nodes; it creates two adjacent conflicts.
- The override is optional, not mandatory.
- Use `long` in Java because the sum can be large.
