## Idea (Simple)
We build the binary schedule from left to right.

At every position, the schedule may end with:
- `0`
- `1`

The important rule is that `1` cannot come after another `1`.
Also, some maintenance positions cannot contain `1`.
So we count schedules separately by their last bit.

## Pattern Recognition Clues
- The problem asks for the number of valid schedules, so this is counting DP.
- Each position has a choice: place `0` or place `1`.
- Whether we can place `1` depends on the previous bit.
- Maintenance slots create a direct restriction on the current position.

## State Discovery
The first simple state a student may try is:

`dp[i] = number of valid schedules of length i`

But this is not enough.
It tells how many schedules exist, but it does not tell how those schedules end.

That missing detail matters because a new `1` cannot follow an old `1`.
So we split the state:

- `endWithZero[i]`: number of valid schedules of length `i` ending with `0`
- `endWithOne[i]`: number of valid schedules of length `i` ending with `1`

This stores exactly the information needed for the next position.

## Pattern
Dynamic Programming with Last Choice State

## Core Insight
- A `0` can follow either `0` or `1`.
- A `1` can follow only `0`.
- If position `i` is a maintenance slot, `endWithOne[i] = 0`.
- The final answer must end with `0`, so return `endWithZero[n]`.

## Invariant
After processing position `i`:
- `endWithZero[i]` stores all valid schedules of length `i` ending with `0`.
- `endWithOne[i]` stores all valid schedules of length `i` ending with `1`.

No counted schedule contains consecutive `1`s or places `1` on a maintenance position.

## Algorithm (with one-line why at important steps)
1. Mark all maintenance positions in a boolean array.
   - Why: we need fast checking for whether `1` is blocked at a position.
2. Initialize position `1`.
   - Why: one-bit schedules are the base cases.
3. Set `endWithZero[1] = 1`.
   - Why: the schedule `0` is always valid.
4. Set `endWithOne[1] = 1` only if position `1` is not blocked.
   - Why: a blocked position cannot contain `1`.
5. For each position `i` from `2` to `n`, compute `endWithZero[i]` from both previous states.
   - Why: `0` can safely follow either previous bit.
6. Compute `endWithOne[i]` from only `endWithZero[i - 1]` if the position is not blocked.
   - Why: `1` cannot follow `1`.
7. Return `endWithZero[n]`.
   - Why: the problem requires the schedule to end with `0`.

## Mental Cheat Sheet
| Item | Meaning |
|---|---|
| Pattern | DP with last choice state |
| State split | count schedules by final bit |
| `endWithZero[i]` | valid schedules of length `i` ending in `0` |
| `endWithOne[i]` | valid schedules of length `i` ending in `1` |
| Blocked position | force `endWithOne[i] = 0` |
| Answer | `endWithZero[n]` |

## Complexity
- Time Complexity: `O(n + m)`
- Space Complexity: `O(n)`
- Auxiliary Space: `O(n)`

## Tiny Dry Run
For `n = 4` and blocked position `{2}`:

| Position | endWithZero | endWithOne |
|---|---:|---:|
| 1 | 1 | 1 |
| 2 | 2 | 0 |
| 3 | 2 | 2 |
| 4 | 4 | 2 |

Answer = `endWithZero[4] = 4`.

Valid schedules ending with `0` are:
- `0000`
- `0010`
- `1000`
- `1010`

## Notes / Pitfalls
- Do not return `endWithZero[n] + endWithOne[n]`; the final bit must be `0`.
- A maintenance position blocks only `1`, not `0`.
- Do not allow `endWithOne[i]` to come from `endWithOne[i - 1]`.
- Apply modulo after addition.
