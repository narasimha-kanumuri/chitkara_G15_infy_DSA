# Smart Stair Memory — Quick Solution

## Idea (Simple)

We need to count all valid jump sequences that reach step `n`.

The climber can use jumps of size `1` or `2`.
But a `2`-step jump cannot come immediately after another `2`-step jump.

So while using recursion, we must remember the current step and the previous jump.
Memoization saves answers for repeated states.

## Pattern Recognition Clues

- We are asked to count all valid ways.
- The same step can be reached through many different recursive paths.
- The next choice depends on the previous jump.
- Repeated recursive calls suggest memoization.

## State Discovery

The first simple state is `ways(step)`.

But `ways(step)` is not enough.
It tells where we are, but it does not tell how we reached there.

The rule depends on the previous jump.
If the previous jump was `2`, the next jump cannot be `2`.

So the final state must store both pieces of information:

- `step`: the current step number
- `lastJump`: the jump used to reach this step

We use:

```text
solve(step, lastJump)
```

This means:

```text
number of valid ways to reach n starting from step,
when the previous jump was lastJump
```

## Pattern

Top-Down Dynamic Programming with Last Choice State

## Core Insight

- From any state, try a `1`-step jump.
- Try a `2`-step jump only if the previous jump was not `2`.
- Cache each `(step, lastJump)` result so it is computed once.

## Invariant

For every recursive state `solve(step, lastJump)`:

- `step` always represents the current position.
- `lastJump` always represents the jump used just before this state.
- The function counts only paths that do not contain consecutive `2`-step jumps.

## Algorithm (with one-line why at important steps)

1. Start recursion from `solve(0, 0)`.
   - Why: the climber starts before taking any jump.
2. If `step == n`, return `1`.
   - Why: one valid sequence has reached the target exactly.
3. If `step > n`, return `0`.
   - Why: a path that crosses the target is invalid.
4. If this state is cached, return the saved value.
   - Why: memoization avoids repeated work.
5. Add the result of taking a `1`-step jump.
   - Why: a `1`-step jump is always allowed.
6. Add the result of taking a `2`-step jump only when `lastJump != 2`.
   - Why: this enforces the no-consecutive-2 rule.
7. Save and return the total.
   - Why: future calls can reuse the same answer.

## Mental Cheat Sheet

| Item | Meaning |
|---|---|
| Pattern | Recursion to memoization |
| State | `(step, lastJump)` |
| Choice | Jump `1` or jump `2` |
| Restriction | Jump `2` is blocked if `lastJump == 2` |
| Base case | `step == n` gives one valid way |
| Invalid case | `step > n` gives zero ways |
| Answer | `solve(0, 0)` |

## Complexity

- Time Complexity: `O(n)`
- Space Complexity: `O(n)`
- Auxiliary Space: `O(n)` for memoization and recursion stack

There are only about `2 * (n + 1)` useful states because `lastJump` only needs to distinguish whether the previous jump was `2` or not.

## Tiny Dry Run

For `n = 3`:

```text
solve(0, 0)
├── jump 1 -> solve(1, 1)
│   ├── jump 1 -> solve(2, 1)
│   │   └── jump 1 -> solve(3, 1) = 1
│   └── jump 2 -> solve(3, 2) = 1
└── jump 2 -> solve(2, 2)
    └── jump 1 -> solve(3, 1) = 1
```

Total valid ways = `3`.

They are:

```text
1 + 1 + 1
1 + 2
2 + 1
```

## Notes / Pitfalls

- Do not cache only by `step`; the previous jump changes which moves are allowed.
- Do not allow a `2`-step jump when `lastJump == 2`.
- Return `0` when `step > n`; crossing the target is not valid.
- Use `BigInteger` because counts can grow large for bigger `n`.
