# Tile Arrangement Memo — Quick Solution

## Idea (Simple)

We cover the corridor from left to right.
At each point, we may place a tile of length `1` or length `2`.
But a length-`2` tile cannot come immediately after another length-`2` tile.
Also, the final tile must be length `1`.

So we use memoized recursion with state `(remaining, prevTwo)`.

## Pattern Recognition Clues

- We are asked to count all valid arrangements.
- Choices repeat for smaller remaining lengths.
- A future choice depends on the previous tile.
- Repeated recursive calls suggest memoization.

## State Discovery

The first simple state is `solve(remaining)`.

But `solve(remaining)` is not enough.
It does not know whether the previous tile was length `2`.

That missing information decides whether another length-`2` tile is allowed now.

So we use:

```text
solve(remaining, prevTwo)
```

Meaning:

```text
number of valid ways to cover remaining length,
when prevTwo tells whether the previous tile was length 2
```

## Pattern

Top-Down Combinatorial DP with Last Tile State

## Core Insight

- A length-`1` tile is always allowed.
- A length-`2` tile is allowed only if the previous tile was not length `2`.
- When no length remains, count the tiling only if the last tile was length `1`.

## Invariant

For every state `solve(remaining, prevTwo)`:

- The already placed prefix is valid.
- `remaining` is the uncovered corridor length.
- `prevTwo` correctly stores whether the last placed tile was length `2`.
- No counted arrangement contains adjacent length-`2` tiles.

## Algorithm (with one-line why at important steps)

1. Start with `solve(n, 0)`.
   - Why: no tile has been placed yet, so the previous tile is not length `2`.
2. If `remaining == 0`, return `1` only when `prevTwo == 0`.
   - Why: this enforces that the final tile must be length `1`.
3. If `remaining < 0`, return `0`.
   - Why: the tiling has crossed the corridor length.
4. If the state is cached, return it.
   - Why: the same state can repeat through different prefixes.
5. Place a length-`1` tile and move to `solve(remaining - 1, 0)`.
   - Why: a small tile is always valid and resets the last-tile flag.
6. Place a length-`2` tile only if `prevTwo == 0`.
   - Why: this prevents adjacent long tiles.
7. Store and return the total.
   - Why: future calls can reuse the answer.

## Mental Cheat Sheet

| Item | Meaning |
|---|---|
| Pattern | Memoized tiling DP |
| State | `(remaining, prevTwo)` |
| `remaining` | corridor length still uncovered |
| `prevTwo` | whether previous tile was length `2` |
| Final rule | count only when last tile was length `1` |
| Answer | `solve(n, 0)` |

## Complexity

- Time Complexity: `O(n)`
- Space Complexity: `O(n)`
- Auxiliary Space: `O(n)` for memo table and recursion stack

There are at most `2 * (n + 1)` states.
Each state checks at most two tile choices.

## Tiny Dry Run

For `n = 4`, valid tilings are:

```text
1 + 1 + 1 + 1
2 + 1 + 1
1 + 2 + 1
```

Count = `3`.

## Notes / Pitfalls

- Do not count tilings that end with tile `2`.
- Do not allow `2 + 2` anywhere.
- Do not cache only by remaining length if previous tile affects the next choice.
- Use `BigInteger` because the count can grow large.
- The empty remaining length is counted only when the last tile is length `1`.
