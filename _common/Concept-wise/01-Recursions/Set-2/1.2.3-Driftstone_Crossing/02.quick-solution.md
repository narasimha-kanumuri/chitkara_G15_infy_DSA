# Driftstone Crossing — Quick Solution

## Idea (Simple)
We try all valid jump choices recursively.

At each stone, the robot can jump `1`, `2`, or sometimes `3` stones forward.

The answer is the minimum energy among all valid future routes.

Because some costs depend on past choices, the recursion must remember the previous jump and whether the `3-step` jump was already used.

## Pattern Recognition Clues
- We need the minimum cost, not the number of routes.
- Each jump moves the robot to a smaller remaining problem.
- A future cost depends on whether the previous jump was `2`.
- The one-time `3-step` jump creates a used / not-used state.

## State Discovery
The first idea is `solve(index)`.

But `solve(index)` is not enough.

It does not know whether the previous jump was `2`, so it cannot apply the fatigue penalty correctly.

It also does not know whether the one-time `3-step` jump was already used.

Final state:
- `index`: current stone
- `lastJump`: previous jump size
- `usedThree`: whether the `3-step` jump has already been used

## Pattern
Optimization Recursion with Constraint State

## Core Insight
- Every route from a stone can be explored by trying allowed jumps.
- The return value should be the minimum energy from the current state to the final stone.
- Extra state is needed only for rules that depend on earlier decisions.

## Invariant
For every call `minEnergy(index, lastJump, usedThree)`, the function returns the minimum additional energy needed to reach the last stone from `index`, while respecting all constraints already recorded in the state.

## Algorithm (with one-line why at important steps)
1. If `index == n - 1`, return `0`.
   - Why: no more energy is needed after reaching the last stone.
2. Start the answer as a very large value.
   - Why: we are minimizing over possible jumps.
3. Try jump `1` and jump `2` if they stay inside the array.
   - Why: these are normal allowed moves.
4. Add height difference as the jump cost.
   - Why: energy depends on height change.
5. If the current jump is `2` and `lastJump` is also `2`, add fatigue penalty.
   - Why: consecutive `2-step` jumps are costly.
6. Try jump `3` only if it has not been used yet.
   - Why: the `3-step` jump is allowed exactly once.
7. Recursively solve the next state and take the minimum.
   - Why: each jump creates a smaller subproblem.

## Mental Cheat Sheet
| Item | Meaning |
|---|---|
| Pattern | Optimization recursion with extra state |
| State | `(index, lastJump, usedThree)` |
| Normal jumps | `1` and `2` |
| Special jump | `3`, only once, double cost |
| Fatigue | add `p` when `lastJump == 2` and current jump is `2` |
| Answer | `minEnergy(0, 0, false)` |

## Complexity
- Time Complexity: `O(3^n)` in the worst case.
- Space Complexity: `O(n)` because recursion depth can reach `n`.
- Auxiliary Space: `O(n)` recursion stack.

## Tiny Dry Run
For heights `[10, 30, 20, 40, 25]` and `p = 7`:

- From stone `0`, jump `2` reaches stone `2` with cost `10`.
- From stone `2`, another jump `2` reaches stone `4` with cost `5`.
- Because two `2-step` jumps are consecutive, add fatigue `7`.
- Total is `10 + 5 + 7 = 22`, and no better route exists.

Answer = `22`.

## Notes / Pitfalls
- Do not use only `index` as the state.
- Add fatigue only on the second consecutive `2-step` jump.
- A `3-step` jump must be blocked after it is used once.
- The base case returns `0`, not the current stone height.
