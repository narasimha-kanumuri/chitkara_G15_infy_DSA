# Locked Steps Cache — Quick Solution

## Idea (Simple)

We count the number of valid ways to reach step `n` from the current step.
From most steps, we may move `1` or `2` steps.
But locked steps cannot be landed on.
If the current step is prime, the next move must be exactly `1` step.

Memoization stores the answer for each step so repeated recursive calls are avoided.

## Pattern Recognition Clues

- The task asks for the number of valid ways.
- Choices repeat from many step positions.
- Locked steps and prime-step rules restrict transitions.
- Repeated calls from the same step suggest memoization.

## State Discovery

The first simple state is:

```text
solve(step)
```

This state is enough.
The locked-step list is fixed.
Prime status is also fixed for every step.

So future choices depend only on the current step.
No extra state like previous jump or coupon usage is needed.

Final state:

```text
solve(step) = number of valid ways to reach n from step
```

## Pattern

Top-Down Counting DP with Constraint Handling

## Core Insight

- If the current step is locked, it contributes `0` ways.
- If the current step is `n`, it contributes `1` valid way.
- If the current step is prime, only the `+1` move is allowed.
- Otherwise, both `+1` and `+2` moves are considered.

## Invariant

For every `solve(step)`:

- The student is currently at `step`.
- No locked step has been accepted as valid.
- Prime-step movement rule is applied before choosing the next move.
- The function returns the exact number of valid ways from `step` to `n`.

## Algorithm (with one-line why at important steps)

1. Mark all locked steps in a boolean array.
   - Why: checking whether a step is locked must be fast.
2. Precompute prime status up to `n`.
   - Why: prime checks are needed during recursion.
3. Start with `solve(0)`.
   - Why: the student starts at step `0`.
4. If `step > n`, return `0`.
   - Why: crossing the target is invalid.
5. If `step` is locked, return `0`.
   - Why: locked steps cannot be landed on.
6. If `step == n`, return `1`.
   - Why: one valid route has reached the target exactly.
7. If the answer is cached, return it.
   - Why: repeated states should be solved once.
8. If `step` is prime, use only `solve(step + 1)`.
   - Why: prime steps force a `1`-step move.
9. Otherwise use `solve(step + 1) + solve(step + 2)`.
   - Why: both normal moves are allowed.
10. Cache and return the result.
    - Why: future calls can reuse it.

## Mental Cheat Sheet

| Item | Meaning |
|---|---|
| Pattern | Memoized counting DP |
| State | `step` |
| Locked step | contributes `0` ways |
| Prime step | only `+1` move allowed |
| Normal step | `+1` and `+2` moves allowed |
| Answer | `solve(0)` |

## Complexity

- Time Complexity: `O(n)`
- Space Complexity: `O(n)`
- Auxiliary Space: `O(n)` for memo table and recursion stack

Each step from `0` to `n` is computed at most once.

## Tiny Dry Run

For:

```text
n = 4, locked = {}
```

Key states:

```text
solve(4) = 1
solve(3) = solve(4) = 1        because 3 is prime
solve(2) = solve(3) = 1        because 2 is prime
solve(1) = solve(2) + solve(3) = 2
solve(0) = solve(1) + solve(2) = 3
```

Answer = `3`.

## Notes / Pitfalls

- Step `1` is not prime.
- Check locked steps before counting a step as a valid target.
- Do not allow `+2` from a prime-numbered current step.
- Do not count paths that move beyond `n`.
- Use `BigInteger` because the count can grow large.
